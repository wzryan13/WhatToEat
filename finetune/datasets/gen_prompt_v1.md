# 任务：为「菜谱 RAG 查询理解」批量生成微调数据（query + 改写 + 标签）

你要为一个中文菜谱检索系统**批量生成微调训练数据**。系统里有一个「查询理解」模块：输入一句用户的做菜需求，输出两样东西——改写后的检索查询 `rewritten_query`，和 Milvus 元数据过滤表达式 `expr`。

本次任务：你要**自己构造 100 条用户做菜 query**，并为每一条同时给出 `rewritten_query` 和 `expr`。也就是 query、改写、标签三样都由你生成。

下面先给出与生产模块**完全一致**的标注规则（任务一改写 + 任务二过滤），你生成的每一条都必须严格遵守这套规则；之后给出生成维度、红线、示例和输出格式。

================ 任务一：查询改写（rewritten_query）================

你是食谱数据库的智能搜索助手。你的任务是将用户的输入优化为一个**清晰、自然且完整**的句子，以便进行语义搜索。

**准则：**
1.  **仅限自然语言：** 不要输出关键词堆砌。重写后的查询必须是一个语法完整的句子或自然的问句（例如"我该如何制作……"或"有哪些……"）。
2.  **严禁幻觉：** 除非用户明确提及，否则不要添加具体的形容词（如"简单的"、"快速的"、"健康的"、"辣的"）。
3.  **澄清但不设限：** 如果查询很模糊（例如"我饿了"），将其重写为请求食物推荐的通用但清晰的句子，除非用户指定，否则不要假设是午餐还是晚餐。
4.  **扩展概念：** 对于推荐类查询，可以适当扩展相关概念以提高检索效果。例如"荤素搭配"可以扩展为"既有肉类又有蔬菜的菜品"。
5.  **保持语气：** 保持礼貌和对话感，与原查询的语言风格相匹配。

**示例：**

-   Original: "我想做点吃的" -> Rewritten: "你能推荐一些适合我做的食谱吗？"
-   Original: "今晚吃啥？" -> Rewritten: "今晚晚餐有什么好的食谱推荐吗？"
-   Original: "有鸡蛋和西红柿，能做什么" -> Rewritten: "用鸡蛋和西红柿可以做什么菜？"
-   Original: "红烧肉做法" -> Rewritten: "如何制作红烧肉？"
-   Original: "来点甜的" -> Rewritten: "给我看一些关于甜点或甜食的食谱。"
-   Original: "有什么荤素搭配的家常菜？" -> Rewritten: "有哪些既有肉类又有蔬菜的家常菜？"
-   Original: "推荐几道清淡的菜" -> Rewritten: "有哪些口味清淡、不油腻的菜品？"

每条只给 1 句重写后的查询，必须是自然语言完整句/问句，不堆砌关键词。

================ 任务二：元数据过滤表达式（expr）================

根据 query，判断是否可以生成一个可直接用于 Milvus `expr` 参数的布尔过滤表达式。
**只有在条件明确、无歧义、不会明显损害召回的情况下，才允许生成过滤表达式；否则必须放弃过滤（expr = "NONE"）。**

【最高优先级原则】
- 元数据过滤是**精确约束**，不是语义理解或推理。
- 只有当用户**明确表达**了可直接映射到元数据字段的条件时，才生成过滤。
- 任何不确定、模糊、需要推断的情况，**一律 expr = "NONE"**。

【允许使用的字段（严格限制）】
你 **只能** 使用以下 3 个 metadata 字段，禁止任何其他字段：
- `category`
- `dish_name`
- `difficulty`
**字段值必须严格来自下方【可用元数据取值】，禁止猜测、扩展或改写。**

【字段使用规则】
1. category —— 仅在用户明确指定菜品大类时使用；不得从口味、场景、食材推断。
2. dish_name —— 仅在用户明确提及具体菜名时使用，用 `LIKE "%菜名%"`；不得从食材或描述推断。
3. difficulty（高风险）—— 仅在用户明确提到难度要求（简单/新手/困难/复杂等）时使用；未明确提及，禁止使用。

【逻辑组合规则】
- 仅在所有条件都高度确定时才用 `and`；确定性有差异时只保留最确定的条件。
- 必要时用括号明确优先级。

【Milvus 过滤表达式参考】
## Milvus 过滤表达式速查
### 一、比较运算符
* `==` 等于   `!=` 不等于   `>` `<` `>=` `<=` 数值比较
### 二、模式匹配 LIKE（字符串）
* `%` 任意长度字符；前缀匹配性能最好
* 前缀 `name LIKE "红烧%"`；后缀 `name LIKE "%鸡"`；包含 `name LIKE "%红烧%"`
### 三、逻辑运算符
* `and` 全部满足；`or` 任一满足；`not` 取反
* 例：`category == "素菜" and difficulty == "简单"`
* 例：`dish_name LIKE "%鸡%" or dish_name LIKE "%鱼%"`

【可用元数据取值】
来源: recipe_chunks
- category (共10个): 早餐、汤类、主食、甜品、饮品、调料、半成品加工、水产、荤菜、素菜
- difficulty (共5个): 入门、简单、中等、较难、困难

================ 红线（违反即作废，务必逐条自检）================

1. **category 只能是这 10 个值**：早餐/汤类/主食/甜品/饮品/调料/半成品加工/水产/荤菜/素菜。
   **绝对不可出现** 川菜、粤菜、湘菜、东北菜、鲁菜、家常菜、川味、广式 等任何枚举外的值——这些一律走 `"NONE"`。
2. **difficulty 只能是这 5 个值**：入门/简单/中等/较难/困难。
3. **以下情形必须 `expr = "NONE"`（弃权）**：
   - 菜系/地方菜（川菜、粤菜、东北菜、四川风味……库里没有菜系字段）
   - 口味词（辣、清淡、下饭、酸甜、开胃、解腻……无对应字段）
   - 临时否定（不辣、不油腻、别太麻烦、别用太多调料）
   - 食材（排骨、鸡蛋、虾、五花肉……**不得从食材推 category，"虾"≠水产**）
   - 纯模糊/场景（今晚吃啥、随便推荐、快手菜、减脂餐、下酒菜、拿手菜）
   - J 拼接形态（原始意图模糊，菜名只帮检索）
4. `expr` 永远是一个**字符串**：要么是合法 Milvus 表达式，要么字面字符串 `"NONE"`。**禁止 null、空串、其他形式。**
5. 表达式内字段值用双引号，作为 JSON 字符串时转义，例如 `"expr": "dish_name LIKE \"%红烧肉%\""`。

================ 多样性与防重复 ================
- 所有 query 必须是**「在家做菜/找菜谱」语境**（例如"做/自己做/菜谱/教我做/用X能做什么/怎么做/想吃点酸甜口的,有什么菜谱推荐“），不要"找餐厅"语境（"想吃/附近/哪里有/推荐个店"）。
- query 要口语、自然、句式多样，避免重复套路；同一维度内不要近义复述。
- 不要照搬常见示范句，尽量造**新的、没出现过的**说法（不同菜名、不同食材、不同场景、不同口吻）。
- A 维度多换菜名（覆盖荤/素/汤/主食/甜品/水产/早餐等不同品类的具体菜名）；H 维度多换食材；E 维度覆盖川/粤/湘/鲁/东北/江浙/闽等。
