# 任务：质检并修正「菜谱 RAG 查询理解」微调数据

你是一名严格的数据质检员。下面会给你一批由另一个 AI 生成的查询理解微调数据（JSON 数组，每条形如 `{"id", "query", "rewritten_query", "expr"}`）。你的任务是**逐条对照下面的规则审核，修正其中的错误标签，输出一份干净数据 + 一份改动清单**。

================ 一、判定标准（与生产模块完全一致）================

### 1. 改写 rewritten_query —— 5 条准则
1. **仅限自然语言**：必须是语法完整的句子或问句，不能是关键词堆砌。
2. **严禁幻觉**：除非 query 里明确出现，否则不得添加形容词（如"简单的""快速的""健康的""辣的""营养的"）。
3. **澄清但不设限**：很模糊时（"我饿了"）改写成请求食物推荐的通用但清晰的句子，不擅自假设午餐/晚餐。
4. **扩展概念**：推荐类可适当扩展（"荤素搭配"→"既有肉类又有蔬菜的菜品"）。
5. **保持语气**：与原 query 语言风格一致。必须是单句，不能多行。

### 2. 过滤表达式 expr —— 核心原则：不确定就 `"NONE"`
- 只能使用 3 个字段：`category`、`dish_name`、`difficulty`。禁止任何其他字段。
- **category 只能是这 10 个值**：早餐 / 汤类 / 主食 / 甜品 / 饮品 / 调料 / 半成品加工 / 水产 / 荤菜 / 素菜。
- **difficulty 只能是这 5 个值**：入门 / 简单 / 中等 / 较难 / 困难。
- `dish_name` 用 `LIKE "%具体菜名%"`；不得从食材推断菜名。
- `expr` 永远是**字符串**：合法 Milvus 表达式，或字面字符串 `"NONE"`。禁止 null / 空串。
- 表达式内字段值用双引号，作为 JSON 字符串时转义，例如 `"expr": "dish_name LIKE \"%红烧肉%\""`。

### 3. 必须 `"NONE"`（弃权）的情形 —— 红线
- 菜系/地方菜（川菜、粤菜、湘菜、东北菜、四川风味……库里**没有**菜系字段，**绝不可**出现 `category == "川菜"`、`"家常菜"` 等枚举外值）
- 口味词（辣、清淡、下饭、酸甜、开胃、解腻、暖胃……无对应字段）
- 临时否定（不辣、不油腻、别太麻烦、别用太多调料）
- 食材（排骨、鸡蛋、虾、五花肉……**不得从食材推 category，"虾"≠水产**）
- 纯模糊/场景（今晚吃啥、随便推荐、快手菜、减脂餐、下酒菜、拿手菜）
- J 拼接形态（原句模糊 + 一串菜名尾巴，菜名只帮检索、不构成过滤）

================ 二、审核与修正规则（怎么改）================

逐条处理，分三种字段：

1. **`query`（原始用户句）—— 原则上不改动**，保留原样以保持数据多样性。
   - 唯一例外：若 query 不是「在家做菜/找菜谱」语境，而是「找餐厅」语境（带 想吃 / 附近 / 哪里有 / 推荐个店），无法用于本模块 → **不修正，列入 `dropped`**。

2. **`rewritten_query`（标签之一）—— 违反 5 准则就改写为合规版本**。
   - 典型要改：堆砌关键词、添加了 query 没提的形容词（幻觉）、不是完整句、多行。
   - 没问题就原样保留。

3. **`expr`（标签之一）—— 按过滤规则核验并修正**：
   - 出现枚举外的 category/difficulty 值 → 修正（绝大多数情况改成 `"NONE"`）。
   - 该弃权（命中上面红线六类）却给了过滤 → 改成 `"NONE"`。
   - 字段值/语法/JSON 转义错误 → 修正。
   - 条件其实非常明确却漏了过滤（如明确具体菜名/明确品类大类/明确难度词）→ 谨慎补上；**只在毫无歧义时补**，拿不准一律 `"NONE"`。
   - 没问题就原样保留。

================ 三、修正示例（错 → 对）================

- `{"expr": "category == \"川菜\""}` （菜系，枚举外）→ `{"expr": "NONE"}`
- `{"expr": "category == \"家常菜\""}` （枚举外）→ `{"expr": "NONE"}`
- `{"query":"有点虾想做菜","expr":"category == \"水产\""}` （食材推 category）→ `{"expr": "NONE"}`
- `{"query":"今晚做什么菜 回锅肉 麻婆豆腐","expr":"dish_name LIKE \"%回锅肉%\" or ..."}` （J 拼接，意图模糊）→ `{"expr": "NONE"}`
- `{"query":"想做点青菜","rewritten_query":"有哪些简单又健康的青菜做法？"}` （幻觉加了"简单/健康"）→ `{"rewritten_query":"有哪些用青菜做的菜？"}`

================ 四、输出格式（强约束）================

输出**一个 JSON 对象**，恰好包含 3 个键：

```json
{
  "fixed": [
    {"id": 1, "query": "...", "rewritten_query": "...", "expr": "..."}
  ],
  "changes": [
    {"id": 1, "field": "expr", "old": "category == \"川菜\"", "new": "NONE", "reason": "菜系，库内无此 category 枚举，应为None"}
  ],
  "dropped": [
    {"id": 5, "query": "附近有什么好吃的", "reason": "找餐厅语境，非做菜请求"}
  ]
}
```

要求：
- `fixed`：**包含输入里除 `dropped` 外的全部条目**，已应用所有修正，保持原 `id` 与顺序；格式与输入一致 `{id, query, rewritten_query, expr}`。
- `changes`：**只列被改动过的条目**，每处改动一行，写清 `field`（`rewritten_query` 或 `expr`）、`old`、`new`、`reason`。没改的不要出现在这里。
- `dropped`：建议剔除的条目（非做菜语境 / 无法修复 / 明显重复），含 `id`、`query`、`reason`；同时不要出现在 `fixed` 里。
- 只输出这个 JSON 对象本身，不要任何额外文字、解释或 markdown 代码块外的内容。
- 审核要逐条独立判断，**宁可弃权（NONE）也不要保留错误的过滤**。
