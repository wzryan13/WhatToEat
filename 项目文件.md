# 一、目标定义

这个项目当前已经有：

1. `thread` 级别的流程状态：`DietState`
2. `checkpointer` 级别的中断恢复：`MemorySaver`
3. `conversation_history` 级别的对话上下文

但它还没有真正的生产级“用户记忆系统”。

生产级记忆系统的目标不是“把用户所有话都存下来”，而是：

1. 让系统跨会话记住稳定偏好
2. 让系统在当前任务中复用短期上下文
3. 让推荐结果随历史行为逐步个性化
4. 让记忆可控、可删、可解释、可审计


# 二、要解决的核心问题

## 1. 现在缺少的能力

当前系统的问题主要有：

1. `MemorySaver` 只能恢复同一个 thread 的状态，不等于用户长期记忆
2. `thread_id` 在 `main.py` 中每次启动随机生成，无法稳定绑定用户
3. `conversation_history` 只保存部分消息，不能作为可靠记忆来源
4. 没有区分“一次性条件”和“长期偏好”
5. 没有记忆写入策略，容易误记
6. 没有冲突处理，容易被新旧偏好覆盖错
7. 没有记忆治理能力，生产上风险较高

## 2. 生产级必须分层

记忆必须至少拆成四层：

1. `Checkpoint Memory`
   用于 graph 中断恢复、追问恢复

2. `Session Memory`
   用于一次任务或短时间窗口内的上下文复用

3. `User Profile Memory`
   用于跨会话的稳定画像

4. `Episodic / Event Memory`
   用于保存历史推荐、用户反馈、拒绝原因等可检索事件


# 三、推荐的记忆分层设计

## 1. Checkpoint Memory

职责：

1. 支撑 `clarify -> resume`
2. 保存 graph 当前状态
3. 只面向流程恢复，不面向用户画像

实现建议：

1. 开发环境可用 LangGraph 自带内存 checkpointer
2. 生产环境改成持久化 checkpointer
3. `thread_id` 只代表会话线程，不代表用户身份

## 2. Session Memory

职责：

1. 记住本轮找饭任务里的临时约束
2. 记住用户刚补充的位置、预算、场景
3. 在几十分钟到数小时内有效

适合存的字段：

1. `active_city`
2. `active_location_text`
3. `active_radius`
4. `active_budget_range`
5. `active_negative_conditions`
6. `last_clarification_question`
7. `last_result_summary`
8. `expires_at`

不适合长期写入的字段：

1. “我现在在公司楼下”
2. “今天想吃火锅”
3. “这次预算 50”

## 3. User Profile Memory

职责：

1. 记住稳定、可复用、跨会话有效的信息
2. 在 intent parser 和 rerank 阶段优先注入

适合长期保存的字段：

1. `dietary_restrictions`
2. `allergies`
3. `disliked_ingredients`
4. `disliked_flavors`
5. `favorite_cuisines`
6. `budget_preference`
7. `default_city`
8. `frequent_areas`
9. `meal_time_patterns`
10. `preference_notes`

每个字段都建议带：

1. `value`
2. `source`
3. `confidence`
4. `updated_at`
5. `last_seen_at`

## 4. Episodic / Event Memory

职责：

1. 保存“发生过什么”
2. 为后续推荐提供案例级参考
3. 让系统不止知道“用户是谁”，还知道“用户最近怎么选”

适合保存的事件：

1. 某次推荐给了什么
2. 用户最终采纳了什么
3. 用户拒绝的原因是什么
4. 多次出现的反馈模式是什么

典型字段：

1. `event_type`
2. `user_id`
3. `thread_id`
4. `session_id`
5. `city`
6. `query_snapshot`
7. `constraints_snapshot`
8. `recommended_poi_ids`
9. `chosen_poi_id`
10. `rejected_reasons`
11. `importance`
12. `created_at`


# 四、你这个餐饮场景里，什么应该记，什么不应该记

## 1. 应该记

1. 明确忌口
   例：不吃牛肉、海鲜过敏、乳糖不耐受

2. 稳定偏好
   例：更喜欢日料、偏清淡、喜欢安静适合约会的店

3. 稳定预算
   例：平时人均 50-100，更常接受 150 内

4. 稳定地理偏好
   例：常在浦东陆家嘴附近找饭，常住成都

5. 明确指令型偏好
   例：以后优先推荐连锁、以后不要推荐太辣

## 2. 不应该直接记成长期画像

1. 一次性的时间条件
   例：今天夜宵、明天中午

2. 当前 GPS 或临时位置
   例：我现在在国金中心门口

3. 单次预算
   例：今天预算 30

4. 没有确认过的隐式猜测
   例：用户这次点了日料，不等于长期喜欢日料

## 3. 需要累计证据再升级为长期偏好的信息

1. 常选菜系
2. 常选商圈
3. 对价格敏感程度
4. 对营业时间的关注程度
5. 对环境场景的偏好


# 五、生产版 graph 流程

建议把 graph 从：

`intent_parser -> search/filter/rerank -> result_formatter`

扩成：

`memory_read -> intent_parser -> route -> search/filter/rerank -> result_formatter -> memory_write`

## 1. memory_read 节点

输入：

1. `user_id`
2. `thread_id`
3. `session_id`
4. `user_input`

职责：

1. 读取 `UserProfile`
2. 读取未过期 `SessionMemory`
3. 读取最相关的 `EventMemory`
4. 生成一个“本轮可注入记忆摘要”
5. 回填到 state

输出到 state 的建议字段：

1. `user_id`
2. `session_id`
3. `memory_profile`
4. `memory_session`
5. `memory_events`
6. `memory_context_summary`

## 2. intent_parser 阶段

这里不要只看 `conversation_history`，而要同时参考：

1. 最新输入
2. 当前 thread 历史
3. session 记忆
4. 用户稳定画像

目标：

1. 补全位置
2. 补全默认预算
3. 补全负向条件
4. 避免每次都重复问用户

例子：

如果用户说“今天想吃清淡点的”，而 profile 里已经有：

1. 默认城市 = 上海
2. 不吃辣
3. 常见预算 = 100 内

那么 intent parser 可以直接补全更多约束，而不是重新追问。

## 3. search / filter 阶段

这里建议只使用“高置信、强约束”的记忆：

1. 默认城市
2. 强制忌口
3. 明确价格上限偏好

不要把弱偏好直接变成硬过滤，否则会过筛过度。

## 4. rerank 阶段

这里是长期记忆最有价值的地方。

推荐注入：

1. 用户喜欢的菜系
2. 用户不喜欢的口味
3. 用户偏爱的价格带
4. 用户常见用餐场景
5. 最近被拒绝的原因

做法：

1. 把结构化画像转成简明 profile summary
2. 把最相关的 3-5 条事件摘要拼到 rerank prompt
3. 告诉模型哪些是硬约束，哪些是软偏好

## 5. result_formatter 阶段

建议在结果输出中保留“可解释性素材”，但不要把全部 memory 暴露给用户。

比如内部可保存：

1. 本次推荐使用了哪些记忆
2. 哪些记忆影响了排序
3. 哪些记忆被忽略

## 6. memory_write 节点

这是生产落地最关键的一环。

输入：

1. `user_input`
2. `intent_parser` 结构化结果
3. `result_formatter` 输出结果
4. 可能的用户反馈
5. 旧画像与旧 session

职责：

1. 提取新记忆候选
2. 判断是否允许写入
3. 合并到用户画像
4. 写入本轮事件
5. 更新 session memory
6. 记录审计日志


# 六、记忆写入策略

## 1. 先分类，再写入

每条候选记忆都先分成三类：

1. `hard_fact`
   例：我海鲜过敏

2. `soft_preference`
   例：最近更喜欢韩餐

3. `ephemeral_context`
   例：我现在在徐家汇

分类后再决定写到哪里：

1. `hard_fact -> UserProfile`
2. `soft_preference -> UserProfile 或 EventMemory`
3. `ephemeral_context -> SessionMemory`

## 2. 写入触发规则

### 直接写入

满足以下条件可直接写入：

1. 用户明确陈述
2. 与健康、安全、禁忌有关
3. 语义足够稳定
4. 不依赖一次性场景

### 延迟写入

满足以下条件建议先记为事件，再累计提升：

1. 只出现一次的偏好
2. 来源于隐式行为推断
3. 信号较弱

### 不写入

以下情况不写入长期记忆：

1. 当前定位
2. 一次性预算
3. 临时情绪
4. 模糊表达

## 3. 冲突合并规则

举例：

旧画像：
`default_city = 上海`

新输入：
`我最近都在杭州`

不要简单覆盖，建议：

1. 提升 `杭州` 的近期权重
2. 保留 `上海` 的历史事实
3. 如果连续多次在杭州，再更新默认城市

再比如：

旧画像：
`喜欢川菜`

新输入：
`最近别推荐辣的`

建议：

1. 把“不吃辣/近期不吃辣”先写 session
2. 如果多次复现，再修改 profile 的长期口味偏好


# 七、生产存储设计

## 1. 最小生产组合

建议：

1. `Postgres` 存 profile、session、event、checkpoint 元数据
2. `Redis` 存热点 session 和 profile cache
3. `pgvector` 或同类能力存 event embedding

如果一开始不想太重，第一阶段可以先：

1. `Postgres`
2. `JSONB + 索引`
3. 暂时不做向量检索

## 2. 建议的数据表

### users

1. `user_id`
2. `created_at`
3. `updated_at`

### user_profiles

1. `user_id`
2. `profile_json`
3. `profile_summary`
4. `version`
5. `updated_at`

### session_memories

1. `session_id`
2. `user_id`
3. `session_json`
4. `expires_at`
5. `updated_at`

### memory_events

1. `event_id`
2. `user_id`
3. `thread_id`
4. `session_id`
5. `event_type`
6. `event_json`
7. `summary`
8. `importance`
9. `embedding`
10. `created_at`

### memory_audit_logs

1. `audit_id`
2. `user_id`
3. `memory_type`
4. `operation`
5. `before_json`
6. `after_json`
7. `source`
8. `created_at`

## 3. 为什么不建议一开始只用向量库

因为你的核心记忆大部分是强结构化数据：

1. 城市
2. 忌口
3. 预算
4. 菜系偏好
5. 负向条件

这些非常适合结构化读写，不应该全部扔给语义检索。

向量层更适合：

1. 历史案例召回
2. 拒绝原因相似性召回
3. 复杂偏好总结


# 八、生产接口流程

## 1. 入口层

请求到来时，先装配：

1. `user_id`
2. `thread_id`
3. `session_id`
4. `request_id`

其中：

1. `user_id` 是长期身份
2. `thread_id` 是 graph 对话线程
3. `session_id` 是业务会话

这三个必须拆开，不能混用。

## 2. 处理流程

1. API 层接收用户输入
2. 根据 `user_id` 读取 profile
3. 根据 `session_id` 读取 session memory
4. 根据 query 召回相关 events
5. 注入 graph state
6. graph 执行推荐
7. 输出结果
8. 异步写记忆
9. 记录审计和指标

## 3. 为什么建议异步写

因为：

1. 记忆写入不应阻塞主推荐链路
2. 事件抽取和 embedding 会增加延迟
3. 失败时可以重试

实践上可以采用：

1. 主链路同步更新 session
2. 长期画像和事件异步写入

# 十、上线流程建议

2. 明确用户身份模型

产出：

目标：

## Phase 3：Production

目标：

1. 个性化推荐稳定
2. 记忆误写可控

做法：

1. 加事件召回
2. 加异步写入
3. 加 cache
4. 加审计后台
5. 支持用户删除记忆


# 十一、评估指标

生产上至少要看这些指标：

1. `memory_read_hit_rate`
2. `memory_write_success_rate`
3. `profile_merge_conflict_rate`
4. `session_reuse_rate`
5. `recommendation_acceptance_rate`
6. `clarification_reduction_rate`
7. `memory_caused_error_rate`

你这个场景里，最值得追的业务指标是：

1. 追问次数是否下降
2. 推荐采纳率是否上升
3. 用户是否更少重复说明忌口和预算


# 十二、治理与风险控制

生产系统一定要有下面这些能力：

1. 查看当前用户画像
2. 删除某条记忆
3. 禁止写入某类敏感记忆
4. 对健康和过敏类信息保守处理
5. 对模型推断出的偏好设置低置信度

建议的治理策略：

1. 显式输入 > 隐式推断
2. 多次重复 > 单次出现
3. 近期行为 > 很久以前的行为
4. 用户纠正 > 系统历史记录


# 十三、推荐的第一批实现顺序

如果现在开始落地，不建议一口气全做。

建议顺序：

1. 先修正 `user_id / thread_id / session_id` 三者的职责
2. 给 `DietState` 加 memory 字段
3. 新增 `memory_read` 节点
4. 新增 `memory_write` 节点
5. 落一个最小 `UserProfile + SessionMemory`
6. 把 `intent_parser` 和 `llm_rerank` 接上 memory
7. 再补 `EventMemory`
8. 最后补缓存、审计、异步任务、向量检索


# 十四、当前讨论阶段需要先拍板的几个问题

在正式写代码前，建议先把下面这些问题定下来：

1. 用户身份从哪里来，是否已有账号体系
2. session 的生命周期是一次对话、一天，还是自定义超时
3. 哪些饮食信息允许长期保存
4. 是否要支持“删除我的偏好/重置记忆”
5. 当前阶段是否先只做结构化 memory，不上向量检索
6. `memory_write` 是同步还是异步
7. 生产存储是优先 Postgres 还是先 SQLite 过渡


# 十五、结论

对这个项目，生产级记忆系统的正确方向不是“把 LangGraph 的 checkpoint 当记忆”，而是：

1. `thread checkpoint` 负责流程恢复
2. `session memory` 负责短期任务上下文
3. `user profile` 负责稳定画像
4. `event memory` 负责历史行为与案例召回

真正决定效果和可上线性的，不是向量库本身，而是：

1. 记忆分类是否准确
2. 写入策略是否克制
3. 冲突合并是否稳
4. 身份和生命周期是否清晰
5. 是否具备审计、删除和治理能力
